MEMORIA TÉCNICA INTEGRAL: MÈLTIC GMAO 4.0

CAPÍTULO I: INTRODUCCIÓN Y JUSTIFICACIÓN

1.1. Abstract y Resumen Ejecutivo

The transition towards the Industry 4.0 paradigm demands highly cohesive software architectures capable of bridging the critical gap between Field-Level operational hardware (PLCs, sensors) and Enterprise-Level management software (CMMS/ERP). "Mèltic GMAO 4.0" is presented as a state-of-the-art Computerized Maintenance Management System strictly grounded in the concept of the Digital Twin. By engineering a microservices-oriented backend driven by Java Spring Boot, persisted over a hybrid database topology—MySQL for absolute ACID consistency in relational data and MongoDB for high-throughput time-series telemetry—the system overcomes traditional software bottlenecks. High-fidelity client presentation is achieved through a multi-platform Flutter application leveraging the CanvasKit rendering engine to ensure 60FPS fluid charting without RenderFlex Overflows. Security is enforced through an autonomous RFID hardware layer interfaced with a Wiegand-to-HTTP Controllino Mega polling service, creating an air-gapped authentication perimeter governed by JWT-based Role Control (RBAC). The overarching directive is real-world applicability: driving down Mean Time To Repair (MTTR) while elevating Overall Equipment Effectiveness (OEE).

A nivel de mercado hispanohablante, la digitalización de plantas de producción adolece de una brecha tecnológica insalvable para las pequeñas y medianas empresas (PYMES). Los grandes Sistemas de Gestión de Mantenimiento Asistido por Ordenador asumen que los usuarios dispondrán de licencias corporativas astronómicas y formación extenuante. Mèltic GMAO 4.0 desmitifica este escenario, proveyendo un entorno integral nativo que no solo digitaliza el flujo de papeles diarios del mantenimiento de fábrica, sino que conecta físicamente de manera retroactiva con la maquinaria a través de sus controladores lógicos programables (PLCs). Gracias al uso de la virtualización con Docker Compose y redes internas aisladas, el Gemelo Digital se levanta paramétricamente de forma agnóstica al sistema operativo host, permitiendo la conmutación entre un entorno web para ingenierías de proceso y aplicaciones nativas de movilidad para operarios que operan bajo guantes protectores en zonas geolocalizadas hostiles.

1.2. Justificación Tecnológica: El Gemelo Digital en la Industria 4.0

La gestión del mantenimiento industrial ha navegado crónicamente en el Mantenimiento Reactivo (Correctivo), es decir, actuar en respuesta a fallas catastróficas del equipo. El segundo peldaño evolutivo fue el Mantenimiento Preventivo estructurado en lapsos matemáticos de tiempo teóricos (por ejemplo, revisiones en ciclos de 5.000 horas). No obstante, el desgaste empírico de un cojinete expuesto a 300°C constantes difiere radicalmente de uno operando a 21°C. Aquí reside el motivo primario de existencia del Mantenimiento Basado en Condición (CBM). Un sistema software ciego a estas variables físicas es obsoleto en la actual Cuarta Revolución Industrial.

La justificación técnica de interconectar el hardware al código de Mèltic recae en la creación del "Gemelo Digital". El código desarrollado dentro de la carpeta Backend/src/main/java no se limita a ser un CRUD (Create, Read, Update, Delete). Transforma una señal analógica SPI en formato JSON estandarizado, proveyendo un control en la nube (Edge Computing). Se ha elegido Spring Boot (Java) como núcleo computacional debido a su hegemonía en el control robusto de subprocesos multihilos (Threading). Frente a la naturaleza asíncrona de hilo único de Node.js, donde un ciclo while forzado por el monitoreo incesante de cientos de nodos IoT bloquearía la recepción de peticiones web de otros clientes, el embebido Tomcat de Java despacha el Polling ininterrumpido en *Executor Services* aislados que no asfixian la CPU primaria.

CAPÍTULO IV: SEGURIDAD LOGÍSTICA Y GESTIÓN MATRICIAL DE ACCESOS (RBAC)

Para erradicar intrusiones estáticas, se rechaza la autenticación basada en estado del servidor (Server-Side Sessions JSESSIONID clásicas de JSP o PHP) que se apoya en cookies de dominio fácilmente secuestrables y costosas de sincronizar. El mecanismo de resiliencia nuclear implementado en la clase abstracta JwtAuthenticationFilter basa su filosofía en un modelo Stateless Criptográfico Multi-Capa.

4.1. Fundamentación HMAC-SHA256, RBAC y Rolado

El ecosistema define el Rolado Estricto de Acceso Base (RBAC): `ADMIN`, `JEFE`, y `TECNICO`. La cadena secreta (Secret Key) del sistema Spring Boot firma un documento JSON que atesora internamente el ID numérico del operario de máquina y su escala jerárquica de permisos. Las solicitudes que intentan destruir o editar datos críticos pasan obligatoriamente por el SecurityFilterChain. Si un rol de técnico ataca un endpoint protegido con @PreAuthorize("hasAuthority('ADMIN')"), se emite una negación sin acceder siquiera a la base de datos.

4.2. Historian Service: Colisión Evitada (Debouncing) Wiegand - JWT

Al utilizar un lector RFID SPI conectado a un Controllino, dejar la tarjeta encima más de medio segundo provocaría que la placa lanzara cinco peticiones seguidas al servidor por rebote eléctrico. Para evitar docenas de Tokens JWT basura y desbordar la pila de autenticación, el frontend en Dart ancla *Cancellation Tokens* a través de temporizadores asíncronos (`_rfidTimer?.cancel()`), que funcionan como Debounce Filters. Además, el protocolo unifica hardware y software: El firmware del Hardware de C++ desencadena el Pipeline de Java EE autorizando e inutilizando el input manual al instante.

CAPÍTULO V: DOCUMENTACIÓN DE API (SWAGGER Y CONTRATO DE INTEROPERABILIDAD)

Una de las paradojas catastróficas de los proyectos que implementan frameworks totalmente separados (p.ej. Flutter en Dart y Spring Boot en Java) reside en la rotura silenciosa y constante de la serialización de datos. Si un desarrollador en el backend altera el nombre de una propiedad de "estado_ot" a "estadoOrden", el cliente móvil lanza ParseExceptions no controladas. Mèltic GMAO soluciona este abismo integrando OpenAPI 3.0 mediante Swagger UI.

5.1. Implementación Dinámica en Spring Boot

A diferencia de los engorrosos manuales PDF de APIs de hace una década, la documentación en Mèltic nace fusionada al código fuente (Code-First Approach). A través de las anotaciones @Operation(summary = "...", description = "...") dispuestas por encima de los métodos HTTP en los Controllers, el motor de `springdoc-openapi` inspecciona mediante Reflexión y Generics la estructura exacta de Clases Java (DTOs) inyectada en tiempo de compilación. El resultado es el despliegue automático del servidor en la ruta local /swagger-ui/index.html.

5.2. El Contrato de Interoperabilidad (Swagger como Oráculo Técnico)

El contrato OpenAPI 3 actúa como la "Fuente Unica de Verdad" (Single Source of Truth) del sistema industrial. Gracias a este enfoque lógico, Flutter no tiene que "adivinar" el esquema de tipos del servidor. Swagger estipula rigurosamente los tipos exactos, límites de tamaño (length constraints) e instanciación JWT requeridas. Justificamos su criticidad en el hecho de que, gracias a él, el frontal en Dart puede autogenerar o emparejar rígidamente modelos como el TelemetryModel.fromJson(Map) asegurando de manera infalible que un Int provisto por MongoDB a través de Spring jamás sea transformado accidentalmente en una String rompiendo los cálculos analíticos del OEE.

5.3. Validación y Pruebas Previas a la Inyección Cliente

La integración de un panel Swagger excede el mero beneficio documental; se torna en un centro de Pruebas Unitarias Funcionales (Sandbox). Provee al ingeniero de desarrollo la facultad de insertar el Bearer Token JWT en el candado superior e invocar directamente una sentencia Execute contra el endpoint /api/telemetry o /api/ot. Esto disocia de manera brutal los fallos de red. Si una orden de trabajo (OT) no se actualiza en el Panel de Android, gracias a Swagger podemos aislar la variable instigante: si el botón `Execute` del panel web devuelve un HTTP Status 200 con el Response Body esperado, el defecto debe atribuirse inexorablemente a la capa lógica de Flutter; si en contrapartida arroja un error interno HTTP Status 500, el debug recae en la consola de la JVM, permitiendo una iteración ágil y libre de incertidumbre arquitectónica dentro de la Industria 4.0.
