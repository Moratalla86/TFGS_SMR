4. ABSTRACT

This project documents the analysis, development and deployment of Mèltic GMAO 4.0, a Computerised Maintenance Management System (CMMS) designed for small and medium-sized industrial companies. The system enables maintenance teams to manage the full lifecycle of work orders, monitor equipment status in real time, authenticate technicians using RFID cards, and produce digital reports with signature and photo evidence.

The backend is developed with Spring Boot 3 (Java 17) and exposes a REST API secured with JSON Web Tokens (JWT) and role-based access control (RBAC). Relational data — users, machines and work orders — is stored in MySQL 8.0, while sensor time-series data is stored in MongoDB. The frontend is built with Flutter, which compiles a single shared codebase into Android, Windows Desktop and Web applications. The full system is containerised with Docker to ensure consistent and reproducible deployments.

The result is a functional, multi-platform application that replaces paper-based maintenance workflows with a centralised digital platform, improving traceability and reducing response times in industrial environments.

5. RESUMEN Y OBJETIVOS

El presente Proyecto de Fin de Ciclo documenta el análisis, diseño y desarrollo de Mèltic GMAO 4.0, una aplicación multiplataforma de gestión de mantenimiento industrial orientada a pequeñas y medianas empresas del sector productivo. El sistema nace como respuesta a la dependencia que aún tienen muchas plantas de producción de procesos en papel o basados en hojas de cálculo, que dificultan el seguimiento de averías, la asignación de técnicos y el cumplimiento de los planes de mantenimiento preventivo.

5.1. Objetivos del Proyecto

El objetivo principal es desarrollar un sistema digital centralizado, funcional y desplegable en un entorno real, que cubra de forma integral el ciclo de mantenimiento industrial. Para ello se han definido los siguientes objetivos específicos:

  • Gestionar el ciclo de vida completo de las Órdenes de Trabajo (creación, asignación, ejecución y cierre con firma digital).
  • Identificar a los técnicos de forma segura mediante tarjetas RFID y credenciales con control de acceso por roles (RBAC).
  • Monitorizar en tiempo real los datos de los equipos industriales recibidos desde controladores PLC.
  • Generar informes PDF con firma digital y evidencia fotográfica directamente desde la aplicación móvil.
  • Planificar el mantenimiento preventivo mediante un calendario interactivo con órdenes programadas.
  • Desplegar el sistema completo en contenedores Docker para facilitar su instalación y mantenimiento.

5.2. Motivación

La digitalización del mantenimiento industrial sigue siendo un reto para muchas PYMEs, que no disponen de presupuesto para los grandes sistemas GMAO comerciales. Mèltic GMAO 4.0 ofrece una alternativa de código abierto, desarrollada con tecnologías actuales del sector — Spring Boot, Flutter y Docker — que puede implantarse sin costes de licencia y adaptarse a distintos entornos de producción.

6. ANTECEDENTES

Las plantas de manufactura enfrentan diariamente fallos mecánicos y tiempos de caída (Downtime) que afectan a su rentabilidad. Los sistemas de gestión clásicos, como SAP PM o herramientas basadas en Excel, operan en silos aislados que no escuchan a las máquinas y requieren formación técnica avanzada. El estado del arte evidenció la necesidad de plataformas unificadas Mobile-First con capacidades IoT integradas (Oracle Corporation, 2023).

Se propone una arquitectura cliente-servidor desacoplada. El backend en Java Spring Boot 3 gestiona la lógica de negocio y expone una API REST securizada con JWT, persistiendo datos relacionales en MySQL 8.0 (usuarios, OTs, activos) y datos de telemetría temporal en MongoDB. El frontend en Flutter/Dart compila un único código fuente a tres plataformas: Android, Windows y Web, mientras que un módulo de hardware en Python simula la entrada de PLCs durante el desarrollo (Oracle Corporation, 2023).

10. DESARROLLO TÉCNICO Y ARQUITECTURA

10.1. Diseño de Interfaces e Interacción (Frontend Flutter)

Mèltic GMAO adopta un sistema de diseño basado en Dark Mode industrial con los colores Space Cadet (#1E293B) y Neon Cyan (#00F0FF), definidos en la clase IndustrialTheme. Este enfoque reduce la fatiga visual del operario y proporciona un alto contraste en entornos industriales con iluminación variable.

10.1.1. Paradigma Multiplataforma (Single Codebase): Web, Desktop y Móvil

Al adoptar Flutter, Mèltic consolida un único código fuente que se compila a tres destinos sin duplicar lógica ni mantenimiento. El mismo lib/ genera un APK Android para técnicos de campo, un binario .exe Windows para puestos fijos, y una webapp servida por NGINX. La gestión de estado se implementa mediante el patrón StatefulWidget nativo de Flutter, con setState() para actualizaciones locales y el servicio singleton GlobalTelemetryHistorian (que extiende ChangeNotifier) para el estado compartido de telemetría.

10.2. Seguridad Logística y Gestión Matricial (RBAC)

Mèltic rechaza la autenticación basada en sesiones de servidor (JSESSIONID clásico), adoptando un modelo Stateless JWT. El servidor nunca almacena sesión: cada petición HTTP incluye un Bearer Token firmado con HMAC-SHA256, que el filtro TokenAuthFilter valida e inyecta en el SecurityContextHolder de Spring (Spring Framework, 2023).

10.2.1. Implementación del Filtro de Autenticación

El filtro TokenAuthFilter extiende OncePerRequestFilter, garantizando que cada petición HTTP es interceptada exactamente una vez. El fragmento siguiente muestra el mecanismo central de validación:

// TokenAuthFilter.java – Extracto del filtro de autenticación JWT
@Component
public class TokenAuthFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain filterChain)
            throws ServletException, IOException {

        final String authHeader = request.getHeader("Authorization");

        if (authHeader != null && authHeader.startsWith("Bearer ")) {
            String token = authHeader.substring(7);
            Usuario usuario = tokenService.getUsuarioByToken(token);

            if (usuario != null && SecurityContextHolder.getContext()
                    .getAuthentication() == null) {
                // Construir UserDetails con el rol del usuario
                String rol = usuario.getRol().replace(" ", "_").toUpperCase();
                UserDetails userDetails = User.withUsername(usuario.getEmail())
                        .password("").roles(rol).build();

                UsernamePasswordAuthenticationToken authToken =
                    new UsernamePasswordAuthenticationToken(
                        userDetails, null, userDetails.getAuthorities());
                SecurityContextHolder.getContext().setAuthentication(authToken);
            }
        }
        filterChain.doFilter(request, response);
    }
}

10.3. Documentación de API (Swagger e Interoperabilidad)

Uno de los riesgos habituales en proyectos con dos lenguajes distintos en cliente y servidor es la rotura silenciosa de contratos: si el backend modifica el nombre de un campo JSON (por ejemplo, de estadoOt a estadoOrden), el cliente Flutter falla en silencio al deserializar la respuesta. Para evitar esta situación, Mèltic integra OpenAPI 3.0 a través de la librería Springdoc OpenAPI, que genera y publica automáticamente la especificación completa de la API REST (OpenAPI Initiative, 2024).

10.3.1. Configuración de OpenAPI con Seguridad JWT

La clase OpenApiConfig registra el esquema de autenticación Bearer JWT directamente en la especificación OpenAPI, de modo que la consola Swagger UI puede enviar el token de autorización en todas las peticiones de prueba:

@Configuration
public class OpenApiConfig {

    @Bean
    public OpenAPI openAPI() {
        final String schemeName = "bearerAuth";
        return new OpenAPI()
                .info(new Info()
                        .title("Meltic GMAO API")
                        .version("1.0"))
                .addSecurityItem(new SecurityRequirement().addList(schemeName))
                .components(new Components()
                        .addSecuritySchemes(schemeName, new SecurityScheme()
                                .name(schemeName)
                                .type(SecurityScheme.Type.HTTP)
                                .scheme("bearer")
                                .bearerFormat("JWT")));
    }
}

10.4. Lógica de Negocio Pura (Backend Transaccional)

10.4.1. Ciclo de Vida de Órdenes de Trabajo (MantenimientoService)

El modelo OrdenTrabajo encapsula el ciclo de vida completo de una intervención: desde su creación (SOLICITADA) hasta su cierre (CERRADA), incluyendo firma digital en Base64, foto de evidencia, checklist en JSON y PDF del reporte. El servicio gestiona las transiciones de estado garantizando integridad referencial:

// MantenimientoService.java – Cierre de OT con firma y reporte PDF
@Service
public class MantenimientoService {

    public Optional<OrdenTrabajo> cerrarOT(Long id, Map<String, String> payload) {
        return ordenTrabajoRepository.findById(id).map(ot -> {
            ot.setEstado("CERRADA");
            ot.setFechaFin(LocalDateTime.now());
            // Persistir firma del técnico (Base64 del canvas táctil)
            Optional.ofNullable(payload.get("firmaTecnico"))
                    .ifPresent(ot::setFirmaTecnico);
            // Persistir checklist en JSON y evidencia fotográfica
            Optional.ofNullable(payload.get("checklists"))
                    .ifPresent(ot::setChecklists);
            Optional.ofNullable(payload.get("reportePdfBase64"))
                    .ifPresent(ot::setReportePdfBase64);
            return ordenTrabajoRepository.save(ot);
        });
    }
}

10.4.3. Sistema de Alertas y Notificaciones Push (Firebase FCM)

Cuando el servidor detecta una anomalía o se registra un mantenimiento prioritario, el AlertaService persiste la alerta en MySQL y delega el envío push al FcmService, que invoca el SDK de Firebase Cloud Messaging. La deduplicación evita alertas repetidas para la misma máquina y descripción:

// AlertaService.java – Registro de alerta con deduplicación y push
@Transactional
public void registrarAlerta(Long maquinaId, String maquinaNombre,
                             String severidad, String descripcion) {
    List<Alerta> existentes = alertaRepository
            .findByMaquinaIdAndActivaTrue(maquinaId);
    boolean duplicada = existentes.stream()
            .anyMatch(a -> a.getDescripcion().equals(descripcion));

    if (!duplicada) {
        Alerta alerta = new Alerta(maquinaId, maquinaNombre,
                                    severidad, descripcion);
        alertaRepository.save(alerta);
        // Despachar notificación push al dispositivo Android del técnico
        fcmService.enviarPushAlerta(
            "🚨 Alerta Mèltic: " + maquinaNombre,
            severidad + ": " + descripcion
        );
    }
}

10.6. Telemetría en Tiempo Real (GlobalTelemetryHistorian)

El servicio GlobalTelemetryHistorian es un Singleton Flutter que extiende ChangeNotifier y gestiona el ciclo de vida del polling de telemetría para todas las máquinas simultáneamente. Utiliza un SplayTreeMap como buffer circular ordenado temporalmente (clave: timestamp en milisegundos), garantizando acceso O(log n) al punto más reciente y al más antiguo, necesarios para el eje X de la gráfica (Johnson, 2022).

// GlobalTelemetryHistorian.dart – Buffer circular con SplayTreeMap
class GlobalTelemetryHistorian extends ChangeNotifier {
  static final GlobalTelemetryHistorian instance =
      GlobalTelemetryHistorian._internal();

  // Buffers: maquinaId → SplayTreeMap(timestamp → Telemetria)
  final Map<int, SplayTreeMap<int, Telemetria>> _buffers = {};
  static const int maxBufferSize = 7200; // ~2 horas a 1 punto/segundo

  void _trimBuffer(int id) {
    final buffer = _buffers[id];
    if (buffer == null) return;
    while (buffer.length > maxBufferSize) {
      buffer.remove(buffer.firstKey()); // elimina el punto más antiguo
    }
  }

  void startTracking(List<Maquina> maquinas) {
    // Inicializa un buffer por máquina y arranca polling cada 3 segundos
    _pollingTimer = Timer.periodic(const Duration(seconds: 3),
        (timer) => _pollAll());
  }
}

12. EVOLUCIÓN / TRABAJO FUTURO

El proyecto asienta bases sólidas y cumple con los objetivos del ciclo formativo. Las siguientes líneas de evolución natural ampliarían Mèltic hacia un producto comercializable:

12.1. Despliegue con Nodos ESP32 Inalámbricos

Sustituir las placas de prueba por microcontroladores ESP32 con Wi-Fi nativo eliminaría el cableado entre sensores y servidor, simplificando la instalación en planta y abaratando el CAPEX hardware.

12.2. Módulo ERP: Gestión de Almacén y Presupuestos

Vincular cada OT con el consumo de repuestos y piezas, permitiendo generar presupuestos automáticos para aprobación de gerencia. Reduciría la brecha entre mantenimiento técnico y contabilidad financiera.

12.3. Alertas Preventivas Autónomas por Umbrales

Actualmente las alertas se registran manualmente. El siguiente paso sería un motor de reglas en Spring Boot que dispare OTs preventivas automáticas cuando una métrica (p.ej. temperatura) supere un umbral configurable durante más de N minutos, sin intervención humana.

12.4. Módulo de Gestión Económica

Mèltic gestiona el ciclo técnico del mantenimiento, pero no el económico. Una línea de evolución natural sería incorporar un módulo de gestión de costes que vincule cada Orden de Trabajo con su impacto económico real:

  • Control de repuestos y consumibles: registrar las piezas utilizadas en cada OT, descontarlas de un almacén virtual y generar alertas de reposición automática cuando el stock caiga por debajo de un mínimo.
  • Coste por orden: calcular el coste total de cada OT sumando las horas de mano de obra del técnico (extraídas de fechaInicio y fechaFin) más el coste de los materiales consumidos, y almacenar el resultado en la propia entidad OrdenTrabajo.
  • Informes de rentabilidad: ampliar el módulo de KPIs con indicadores económicos — coste medio por avería, ratio coste preventivo/correctivo y presupuesto anual de mantenimiento — que la dirección pueda consultar desde la pantalla de estadísticas.
  • Integración contable: exportar los datos de coste en formato CSV o mediante una API hacia herramientas de contabilidad como FacturaDirecta o ERPs de código abierto como Odoo, cerrando la brecha entre el departamento técnico y el financiero.

Esta mejora convertiría Mèltic en una herramienta de gestión integral, permitiendo a la dirección de la PYME tomar decisiones basadas en datos reales de coste operativo y no solo en indicadores técnicos.

12.5. Integración de Inteligencia Artificial en el Mantenimiento Predictivo

La arquitectura actual de Mèltic ya recoge y almacena en MongoDB series temporales de datos de sensores (temperatura, vibración, presión). Este histórico constituye la materia prima necesaria para entrenar modelos de inteligencia artificial orientados al mantenimiento predictivo, es decir, anticipar el fallo de un equipo antes de que se produzca en lugar de reaccionar a él.

12.5.1. Detección de Anomalías con Machine Learning

El primer nivel de integración consistiría en añadir un microservicio en Python (Flask o FastAPI) que consuma los datos de telemetría almacenados en MongoDB y aplique algoritmos de detección de anomalías. Modelos como Isolation Forest o LSTM (Long Short-Term Memory) pueden aprender el comportamiento normal de cada máquina y generar una alerta automática cuando se detecta una desviación significativa, sin necesidad de configurar umbrales manuales.

El flujo sería el siguiente: el microservicio Python analiza los últimos N puntos del histórico de un sensor, calcula la probabilidad de fallo y, si supera un umbral de confianza, llama al endpoint POST /api/alertas del backend Spring Boot para registrar la alerta y enviar la notificación push al técnico responsable.

12.5.2. Generación Automática de Órdenes de Trabajo Predictivas

Un nivel más avanzado sería conectar el modelo predictivo directamente con el ciclo de OTs: cuando el modelo detecte que una máquina tiene una probabilidad de fallo superior al 80% en las próximas 48 horas, el sistema crearía automáticamente una OT de tipo PREVENTIVA, la asignaría al técnico disponible y la añadiría al calendario de mantenimiento, sin intervención humana.

12.5.3. Asistente de Diagnóstico con IA Generativa

A más largo plazo, la integración de un modelo de lenguaje (LLM) a través de la API de OpenAI o un modelo local como Llama permitiría ofrecer al técnico un asistente de diagnóstico dentro de la propia aplicación Flutter. El técnico podría describir en texto libre los síntomas de una avería y el asistente, entrenado con el historial de OTs cerradas de la empresa, sugeriría las causas más probables y los pasos de resolución más eficaces según la experiencia acumulada en el sistema.

12.6. Conclusiones Generales

Mèltic GMAO 4.0 nació con un objetivo claro: demostrar que es posible construir, con los conocimientos de un ciclo formativo de grado superior, una herramienta de gestión industrial real, funcional y desplegable en un entorno de producción. El resultado obtenido supera ese punto de partida.

El sistema cubre de forma integral el ciclo de mantenimiento industrial: desde la identificación segura del técnico mediante tarjeta RFID, pasando por la emisión y ejecución de órdenes de trabajo con firma digital y evidencia fotográfica, hasta la monitorización en tiempo real de equipos industriales y la planificación del mantenimiento preventivo en un calendario interactivo. Todo ello está respaldado por una API REST documentada con OpenAPI 3.0, desplegada en contenedores Docker y accesible simultáneamente desde Android, Windows y navegador web con un único código fuente Flutter.

Los retos técnicos más significativos del proyecto han sido también los más formativos. La gestión de la concurrencia en el GlobalTelemetryHistorian con buffers circulares SplayTreeMap, la implementación de RBAC granular en Spring Security mediante SecurityFilterChain, la integración de Firebase Cloud Messaging para notificaciones push en Android, y la resolución de errores de layout en distintas resoluciones de pantalla, son problemas reales de la industria del software que han requerido soluciones técnicas documentadas y verificadas.

Las líneas de evolución identificadas — integración de hardware inalámbrico ESP32, módulo de gestión económica vinculado a cada OT, y aplicación de inteligencia artificial para el mantenimiento predictivo — no son adornos teóricos, sino extensiones directas de la arquitectura actual. El histórico de telemetría en MongoDB, los datos de tiempo de OTs y el motor de alertas del backend son la base sobre la que se construirían de forma natural esas mejoras.

El resultado es una aplicación estructurada, probada y documentada que acredita el dominio de las competencias de Acceso a Datos, Programación de Servicios y Procesos, y Programación Multimedia y Dispositivos Móviles, aplicadas a un problema logístico industrial real con impacto económico medible: reducción del tiempo de respuesta ante averías, trazabilidad completa del trabajo técnico y base de datos suficiente para tomar decisiones de mantenimiento basadas en datos.
